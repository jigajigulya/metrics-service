1. Архитектура решения
  Транспортный слой (REST API): 
   `POST /auth` — публичный эндпоинт для генерации JWT-токенов по уникальному `clientID`. Аутентификация на базе алгоритма шифрования HMAC-SHA256 (заменить в проде на RSA).
   `POST /metrics` — защищенный эндпоинт приема метрик. Принимает точные числовые значения и произвольные метаданные (payload).
   `GET /metrics` — защищенный эндпоинт для извлечения агрегированной статистики за интервал дат.
	Слой фильтрации и безопасности (Servlet Filters): Разработана цепочка безопасности Spring Security. Сначала `JwtAuthenticationFilter` валидирует подпись токена 
	и заполняет контекст авторизации, затем `RateLimitingFilter` выполняет контроль лимитов.
	Слой бизнес-логики и СУБД: Сервис считывает данные и аггрегирует в СУБД. Вся логика ошибок централизована в `GlobalExceptionHandler` (`@RestControllerAdvice`).

2. Работа с датами и часовыми поясами (ISO 8601)
 1. Java (`OffsetDateTime`): Входящие параметры `from`, `to` и `timestamp` маппятся в объекты `java.time.OffsetDateTime`, что позволяет сохранять исходное смещение часового пояса
 2. JDBC & Hibernate: В `application.properties` зафиксирована проперти `spring.jpa.properties.hibernate.jdbc.time_zone=UTC`. Она принудительно заставляет драйвер передавать временные метки в сессию базы данных строго в формате UTC, делая приложение независимым от часового пояса машины.
 3. Слой СУБД (`TIMESTAMPTZ`): В схеме таблиц для хранения временных меток выбран тип данных `TIMESTAMP WITH TIME ZONE` (`TIMESTAMPTZ`). При записи PostgreSQL автоматически нормализует любое входящее время к UTC.

3. Реализация Rate Limiting (Redis + Bucket4j)

Контроль частоты запросов реализован на базе алгоритма маркерной корзины с помощью связки библиотеки `Bucket4j` и распределенного кэша `Redis` (через асинхронный драйвер `Lettuce`):

 Использование `Bucket4jLettuce.casBasedBuilder` переносит вычисления лимитов на Redis. Проверка и списание токенов выполняются через атомарные операции. Потоки Java не блокируются.
 Разделение лимитов:
   Глобальный лимит системы: 1000 запросов в секунду (`1s`).
   Клиентский лимит: 10 запросов в секунду (`1s`) для конкретного `clientID` (извлекается из JWT) для защиты от спама одного пользователя.
 Жизненный цикл памяти: Логика очистки неактивных ключей (`ratelimit:client:`) делегирована политике вытеснения самого сервера Redis (`maxmemory-policy allkeys-lru`).
 Если лимит превышен, `RateLimitingFilter` перехватывает запрос до входа в Spring MVC, сериализует JSON ошибки напрямую в `HttpServletResponse` со статусом `429 Too Many Requests` и прерывает выполнение через `return`.

4. Выбор базы данных (PostgreSQL)

СУБД PostgreSQL выбрана как оптимальное решение с точки зрения моего опыта, скорости разработки:

 Согласно ТЗ, метод `GET /metrics` возвращает `count`, `avg`, `min` и `max`. Чтобы эти операции не приводили к полному сканированию таблицы (`Seq Scan`), через построен B-Tree индекс на поле времени.
 
 5. Возможности масштабирования (Highload)

Архитектура сервиса полностью готова к горизонтальному масштабированию:

1. Горизонтальное масштабирование приложения: Так как бэкенд полностью Stateless, мы можем запустить 10 или 50 инстансов приложения за балансировщиком нагрузки. Инстансы не хранят состояние в памяти, а конкурентный доступ к общим счетчикам лимитов атомарно синхронизируется в едином узле Redis.
2. Внедрение брокера сообщений (Kafka / RabbitMQ): При росте нагрузки до 10 000+ RPS, пишем в Kafka отдаем клиенту успешный респонс, обрабатываем бачами и пишем в БД

7. Инструкция по сборке и запуску

Для запуска всего комплекса (бэкенд, PostgreSQL, Redis) на локальном компьютере необходим только установленный Docker Desktop.

В корневом каталоге проекта (рядом с `pom.xml` и `docker-compose.yml`) выполните команду:
docker compose up --build -d



Просмотр логов бэкенда:
Убедиться, что Spring Boot успешно запустился и Tomcat готов принимать трафик, можно командой:

docker logs metrics-backend --tail 50 -f
Постучать в redisЖ
docker exec -it metrics-redis redis-cli
keys * - запросить ключи
Постучать в Postgres
docker exec -it metrics-postgres psql -U metrics_user -d metrics_db
select * from metrics - проверочный запрос на наличие метрик( после Post запроса)
Полная остановка и очистка памяти:

docker compose down -v - остановка контейнера и сброс pgdata
